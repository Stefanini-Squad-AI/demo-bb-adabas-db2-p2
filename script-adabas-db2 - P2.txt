# Prompt

## Produto 1

Será realizada a migração de um programa Natural que atualmente lê dados do ADABAS para um novo fluxo em que o programa Natural fará chamada para um programa COBOL responsável pelo acesso ao DB2.

Execute as tarefas abaixo com base no arquivo NATURAL informado no input e realizando a migração da estrutura ADABAS para DB2.

# Inputs

Arquivos de referência:

- Arquivo NATURAL:
  prg-natural-p2/STFPCS00-P2.txt

- Arquivo ADABAS:
  prg-natural-p2/ADABAS-SOCIOS-P2.txt

# Regras de geração dos arquivos

Todos os arquivos gerados devem seguir a organização abaixo:

- Programas COBOL:
  Pasta Cobol

- Books COBOL:
  Pasta Cobol\src

- Scripts DB2:
  Pasta DB2

- Programas Natural originais:
  Manter em prg-natural-p2

# Objetivo

Converter o fluxo atual:

Natural > ADABAS

Para o novo fluxo:

Natural > COBOL > DB2

# Tarefas

Com base no arquivo ADABAS informado no input, gerar os seguintes artefatos:

1. Um script de criação das tabelas DB2:
   - Destino:
     Pasta DB2

2. Criar um excel no formato igual ao arquivo tabela adabas db2 socio.xlsx.
   - Destino:
     Pasta DB2\Artefatos

3. Book de comunicação entre o programa Natural e o COBOL:
   - Layout compatível com DB2
   - Tipo:
     BOOK LOCAL 

# Programas COBOL

Os programas COBOL devem executar operações DB2 conforme as ações abaixo:

- I = Inclusão
- A = Alteração
- E = Exclusão
- C = Consulta

## Padrão de nomenclatura

Estrutura:

STF + S + C + 00 + [AÇÃO]

Onde:

- STF = Stefanini
- S = Subprograma
- C = COBOL
- 00 = Sequencial
- A/E/I/C = Tipo de operação

## Exemplos

- STFSC00A = Programa de alteração
- STFSC00E = Programa de exclusão
- STFSC00I = Programa de inclusão
- STFSC00C = Programa de consulta

# Regras Gerais

## Programas Natural

- Alterar os programas da pasta prg-natural-p2 removendo o acesso ao adabas pelos novos programas COBOL/DB2.

- Comentários nos acessos antigos ao ADABAS para identificar os pontos de alteração

- Não alterar a quantidade de arquivos NATURAL da pasta original prg-natural-p2.
  Se houver N programas na pasta prg-natural-p2, manter os N programas.

## Comunicação Natural x COBOL

Gerar:
1. Um copybook COBOL reutilizável da(s) entidade(s), compatível com WORKING-STORAGE SECTION, LINKAGE SECTION, FILE SECTION
   e outras áreas do programa na pasta "Cobol\src". Gere um COPYBOOK COBOL usando apenas 1 nível 01 como estrutura principal.  
3. O return code do copybook do Cobol deve refletir os return code do DB2, onde:
   - +000: Registro localizado
   - +100: Registro não localizado
   - +803: Erro insert: chave duplicado
   - Outros return code, o tratamento do erro será realizado genericamente.
   - Esse processamento deve-se refletir nos programas COBOL e Natural para o return code.
2. Uma book LOCAL DATA AREA (LDA) Natural equivalente na pasta "prg-natural\LOCAL". Esse book LOCAL deve ser usado na DEFINE DATA 
   como: Ex: LOCAL USING "Nome_Local".

A estrutura da LDA deve refletir integralmente o copybook COBOL, preservando hierarquia, nomes de campos, tipos, tamanhos, 
OCCURS e REDEFINES, garantindo equivalência funcional entre COBOL e Natural.

- É proibido gerar o código sem o book do Cobol e a Local do natural, esses dois artefatos devem ser gerados
  em todas as execuções.

- O book deve suportar as operações:
  - I
  - A
  - E
  - C

- O book deve conter tamb1ém o respectivo return code.

## Regras de Datas

Quando houver campos de data:

- Utilizar o padrão DB2:

  YYYY-MM-DD

- Não mover diretamente para COBOL campos tipo D do Natural.

- Deve ser realizado:
  - MOVE EDITED
  - utilizando máscara:
    YYYY-MM-DD
  - Quando houver campo hora: utilizar o padrão HH:MM

## Regras do Programa COBOL

O programa COBOL deve conter toda a lógica necessária para acesso ao DB2.

### WORKING-STORAGE SECTION

Declarar SOMENTE:

- Variáveis constantes
- Literais fixos
- Códigos fixos
- Mensagens constantes
- Flags constantes
- Valores imutáveis do programa

### LOCAL-STORAGE SECTION

Declarar SOMENTE:

- SQLCA
- BOOKs de entidades
- BOOK de comunicação Natural x COBOL
- Host variables DB2
- Indicadores DB2
- Cursores
- Estruturas temporárias de execução

### Processamento DB2 Cobol 

- Para operações que realizem múltiplas consultas, leituras ou processamentos repetitivos em uma
  tabela DB2, deve-se obrigatoriamente utilizar cursor para recuperação dos dados.
- Os performs dos programas Cobol gerados não devem conter o nome Periodico na sua estrutura fazendo referencia
  a tabela Adabas migrada a DB2.
  Exemplo: 
  - Perform CARREGA-PERIODICO-CURSOR não deve ser usado. A estrutura do PERFORM deve Refletir o nome da tabela,
    como CARREGA-{Nome_tabela}-CURSOR. Se o nome da tabela demasiado grande, abrevie.
             

- A declaração de todos os comandos de Cursor do DB2 devem estar declaradas na WORKING STORAGE DIVISION.  

### Estrutura interna do programa

Todo programa COBOL gerado deve seguir obrigatoriamente o padrão arquitetural abaixo:

1. A PROCEDURE DIVISION deve conter exclusivamente a sequência principal:
   - PERFORM INICIALIZA
   - PERFORM PROCESSA
   - PERFORM FINALIZA

2. As seções INICIALIZA, PROCESSA e FINALIZA devem existir obrigatoriamente.

3. Responsabilidades de cada seção:
   - INICIALIZA:
     abertura de arquivos, inicialização de variáveis, carga de parâmetros e preparação do ambiente.
   
   - PROCESSA:
     execução das regras de negócio, leitura/processamento de registros e chamadas principais.
   
   - FINALIZA:
     fechamento de arquivos, gravação de totais, exibição de mensagens finais e encerramento do programa.

4. O programa deve finalizar com STOP RUN.

5. Não gerar lógica fora dessas seções principais.

Estrutura esperada:

```cobol
PROCEDURE DIVISION.

    PERFORM INICIALIZA
    PERFORM PROCESSA
    PERFORM FINALIZA

    STOP RUN.

INICIALIZA.
    .

PROCESSA.
    .

FINALIZA.
    .

## Scripts DB2

- Os scripts DB2 devem ser compatíveis com a estrutura identificada nos arquivos NATURAL e ADABAS.
- Campos SUPER do Adabas não devem ser migrados como colunas físicas no DB2; devem ser convertidos
  em índices compostos ou estruturas equivalentes de acesso.

- Migração de estruturas Adabas para DB2 com preservação integral das definições SUPER existentes.
  Toda SUPER do Adabas deve ser reproduzida no DB2 exatamente no mesmo formato, composição e ordem
  de campos, sem criação de novas chaves, índices ou constraints não existentes na origem.

   essa composição deverá ser utilizada como PRIMARY KEY no DB2.

 A modelagem deve permitir crescimento ilimitado
 É PROIBIDO impor limites artificiais
