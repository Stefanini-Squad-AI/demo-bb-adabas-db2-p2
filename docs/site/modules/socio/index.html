Dashboard System Overview Module File Map

Module: socio

Gym/club member (sócio) registration, duplicate RG check, category-based fees, and twelve-month payment schedule.

Current Natural Programs

  • STFPCS00 (prg-natural-p2/STFPCS00-P2.txt) — Online dialog; map MAPSOCIO
  • VERVALOR — Subprogram (not in repo); sets monthly fee array by category

Adabas Dependencies

File / ViewOperationDescription
ADABAS-SOCIOS / SOCIO FIND NUMB-SOCIO-PRINCIPAL EQ #RG-CONSULTA — duplicate detection
ADABAS-SOCIOS / SOCIO STORE New member after INCLUI-SOCIO validations

DDM: ADABAS-SOCIOS

  • Master: RG, name, registration date, category, debt flag, contract end, observations
  • Periodic group PERIODICO-PAGAMENTO: due date, amount, paid flag
  • Descriptor SUPER1 = category + debt indicator (index in DB2, not a column)

Target COBOL Services

  • STFSC00C — Consult member by RG (replaces FIND)
  • STFSC00I — Insert member and payment periodic rows (replaces STORE)

No STFSC00A (alter) or STFSC00E (delete) — not found in Natural source.

Target DB2 Tables

  • TB_SOCIO — Master attributes from non-PE DDM fields
  • TB_SOCIO_PERIODICO_PAGAMENTO — Child rows for payment periods (1:N; no PE flatten)

Business Rules

  1. User may exit with PF-KEY F3.
  2. RG (#RG-CONSULTA) is mandatory for consultation.
  3. If RG already exists, show Sócio já cadastrado and do not insert.
  4. Name, category (1 or 2), and due day (1, 5, 15, 20, or 25) are required on insert.
  5. Registration date set to system date; first of twelve installments marked paid.
  6. Monthly amounts loaded via CALLNAT 'VERVALOR' from category.
  7. Twelve due dates generated by advancing month from selected day-of-month.
  8. Debt indicator and contract-end fields cleared on new member; default observation Novo sócio.

User Story Patterns

Consult before register

As a club attendant,
I want to check whether an RG is already registered,
So that we do not create duplicate member records.

Acceptance criteria:
- Given a valid RG, when the member exists in DB2, then the system shows "Sócio já cadastrado."
- Given a valid RG, when no member exists, then the inclusion flow is offered.
- Adabas FIND is replaced by STFSC00C with return code +100 for not found.

Register new member

As a club attendant,
I want to register a new member with category and payment due day,
So that the member has a twelve-month payment schedule with the first installment marked paid.

Acceptance criteria:
- Validates name, category in {1,2}, due day in {1,5,15,20,25}.
- Persists master row and twelve child payment rows via STFSC00I.
- Fees populated per category (VERVALOR or successor service).
- Returns success message on SQLCODE +000; handles +803 duplicate on concurrent insert.

Decouple persistence

As a modernization engineer,
I want STFPCS00 to call COBOL services instead of Adabas FIND/STORE,
So that all persistence runs against DB2 with standardized return codes.

Acceptance criteria:
- Legacy FIND/STORE commented and replaced by LDA + COBOL call.
- No VIEW OF ADABAS-SOCIOS in production DEFINE DATA.
- Dates exchanged as YYYY-MM-DD per project standards.

Technical Risks

  • External VERVALOR may still use Adabas — document interface before cutover.
  • Map MAPSOCIO not in repo — obtain for UI testing.
  • Natural field OBVS-CLIENTE vs DDM OBVS-SOCIO — normalize in shared books.
  • Twelve-row insert must be atomic in STFSC00I to match single Adabas STORE semantics.
