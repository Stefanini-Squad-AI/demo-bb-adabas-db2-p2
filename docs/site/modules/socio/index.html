Módulo: Sócio (socio)

Gestão de cadastro de sócios com pagamentos periódicos de mensalidade. Migração do arquivo Adabas ADABAS-SOCIOS (FNR 24) para DB2.

Programas Natural Envolvidos

ProgramaArquivoFunção
STFPCS00 prg-natural-p2/STFPCS00-P2.txt Transação online de inclusão de novo sócio (mapa MAPSOCIO)

Área de dados e dependências

  • STFSC00L — LDA em prg-natural-p2/LOCAL/STFSC00L.txt
  • VERVALOR — Subprograma externo para obter valor da mensalidade por categoria
  • MAPSOCIO — Mapa de entrada (RG, nome, categoria, dia vencimento, observação)

Regras de Negócio

RN-01 — RG obrigatório: O campo de consulta/inclusão (#RG-CONSULTA) não pode ser zero.
RN-02 — Verificação de existência: Antes da inclusão, consulta COBOL; +100 permite cadastro; +000 bloqueia com mensagem "Sócio já cadastrado".
RN-03 — Nome obrigatório: Nome do sócio não pode ser em branco na inclusão.
RN-04 — Categoria: Valores aceitos: 1 (Principal), 2 (Dependente). Zero e demais valores rejeitados.
RN-05 — Dia de vencimento: Apenas dias 1, 5, 15, 20 ou 25.
RN-06 — Tarifa: Valores de mensalidade obtidos via CALLNAT VERVALOR conforme categoria.
RN-07 — Primeira parcela: PAGAMENTO-OK(1) = TRUE na inclusão; parcelas 2 a 12 = FALSE.
RN-08 — Calendário de vencimentos: Loop de 12 meses incrementando mês/ano a partir do dia informado.
RN-09 — Data cadastro: Preenchida com data corrente no formato YYYY-MM-DD.
RN-10 — Observação padrão: Se observação vazia, grava "Novo sócio".
RN-11 — Chave duplicada: Return +803 na inclusão quando RG já existe no DB2.
RN-12 — Navegação: F3 encerra programa; apenas tecla ENTR processa consulta.

Escopo de Migração DB2

  • Normalização do PE PERIODICO-PAGAMENTO em tabela filha (proibido flatten em 12 colunas).
  • SUPER1 (CATG-SOCIO + INDI-DIVIDA) → índice IX_SOCIO_SUPER1, sem coluna física.
  • Conversão de campos lógicos Natural para CHAR(1) Y/N no DB2.
  • Limite operacional de 12 pagamentos no programa atual (constante COBOL); modelo DB2 permite crescimento.

Padrão de User Story

US-01 — Consulta de existência de sócio

Como operador de cadastro,
Quero consultar se um RG já está cadastrado,
Para evitar duplicidade antes de incluir um novo sócio.

Critérios de aceite:
- Informar RG no mapa MAPSOCIO e pressionar ENTR
- Se STFSC00C retornar +100, permitir fluxo de inclusão
- Se retornar +000, exibir "Sócio já cadastrado"
- Se retornar erro (+999 ou outro), exibir mensagem com código

US-02 — Inclusão de novo sócio

Como operador de cadastro,
Quero incluir um novo sócio com 12 mensalidades programadas,
Para registrar o associado com calendário de pagamentos correto.

Critérios de aceite:
- Validar nome, categoria (1 ou 2) e dia de vencimento (1,5,15,20,25)
- Obter valores via VERVALOR
- Marcar primeira mensalidade como paga
- Gravar em DB2 via STFSC00I com return +000
- Tratar +803 como chave duplicada

US-03 — Migração de persistência Adabas → DB2

Como arquiteto de modernização,
Quero que STFPCS00 não acesse mais Adabas diretamente,
Para centralizar SQL e transações na camada COBOL.

Critérios de aceite:
- Nenhum FIND/STORE/READ Adabas ativo no fonte Natural
- Comunicação exclusiva via STFSC00L e CALLNAT STFSC00C/STFSC00I
- Return codes alinhados entre Natural, COBOL e SQLCODE DB2

Riscos Técnicos do Módulo

  • Dependência de VERVALOR não documentada neste repositório.
  • Falha após INSERT em SOCIO e antes de concluir 12 INSERTs em SOCIO_PAGAMENTO pode deixar dados inconsistentes.
  • Campos DATA-BAIXA e HORA-BAIXA usados em baixa de contrato — fluxo de UPDATE ainda não implementado.
  • Indicador INDI-DIVIDA e SUPER1 replicados no return da consulta — validar em testes integrados.

Próximos Passos Sugeridos

  1. Inventariar e versionar subprograma VERVALOR.
  2. Implementar ROLLBACK transacional em STFSC00I para falhas parciais.
  3. Executar testes de paridade Adabas vs DB2 em ambiente de homologação.
  4. Expandir escopo com programas de alteração/baixa quando identificados no legado.
